Category 1: Bloaters [4 Marks]
Bloaters are code elements that have grown so large they are hard to work with. They accumulate gradually
as programs evolve and are rarely cleaned up. The five Bloater smells are: Long Method, Large Class, Long
Parameter List, Primitive Obsession, and Data Clumps.
Reference definitions:
• LongMethod—Amethodwithexcessivelines ofcode wherescrolling is required to read it. All logic
crammedinto one function makes it tightly coupled and fragile. Treatment: Extract Method.
• LargeClass —Aclasstrying to do too much, violating the Single Responsibility Principle. As size grows,
cohesion drops. Treatment: Extract Class into focused responsibilities.
• LongParameterList — Amethodsignature withso manyparameters it becomes hard tocall
correctly and reveals that the class is pulling data from too many places. Treatment: Introduce
Parameter Object.
• Primitive Obsession — Using primitive data types (String, int, float) to represent rich domain
concepts instead of small dedicated classes. Treatment: Replace Data Value with Object.
• DataClumps—Thesamegroupoftwotofourvariablesappearing together across multiple methods
or classes, signalling a missing abstraction. Treatment: Extract Class or Introduce Parameter Object.

Category 2: Object-Orientation Abusers [3 Marks]
These smells result from incomplete or incorrect application of object-oriented principles. The four OO
Abuser smells are: Switch Statements, Temporary Fields, Refused Bequest, and Alternative Classes with
Different Interfaces.
• SwitchStatements —Aswitch orchain of if-else that selects behaviour based on a type field. Every
time a newtype is added, every switch must be updated. Treatment: Replace with Polymorphism
(Strategy or State pattern).
• TemporaryFields —Aclass instance variable that is only set and used in one specific scenario,
leaving it null or meaningless the rest of the time. Treatment: Extract Class or Introduce Null Object.
• RefusedBequest —Asubclass inherits methods from a parent but ignores oroverrides most of them,
failing the IS-A test and violating the Liskov Substitution Principle. Treatment: Replace Inheritance
with Delegation or restructure the hierarchy.
• Alternative Classes with Different Interfaces — Two or more classes that do the same thing but have
different method names or signatures, preventing them from being used interchangeably. Treatment:
RenameMethods orExtract Superclass / common interface.

Category 3: Change Preventors [3 Marks]
Change Preventors mean that a single logical change forces modifications in many unrelated places, making
maintenance expensive and error-prone. The three smells are: Divergent Change, Shotgun Surgery, and
Parallel Inheritance Hierarchies.
• Divergent Change —Oneclassmustbe editedfor multiple unrelated reasons (e.g., a change to the
reporting format AND a change to the data validation logic both require editing the same class). This
violates SRP. Treatment: Extract Class to separate responsibilities.
• ShotgunSurgery —Asingle logical change (e.g., changing a tax rate) requires small edits scattered
across many classes. Treatment: Move Method and Move Field to consolidate related behaviour.
• Parallel Inheritance Hierarchies — Every time a subclass is added to one hierarchy, a corresponding
subclass must be added to another. Treatment: Merge hierarchies or use delegation.

Category 4: Dispensables [3 Marks]
Dispensables are things that serve no purpose — their removal would make the code cleaner, smaller, and
easier to read. The five dispensable smells are: Comments (excessive/redundant), Duplicate Code, Lazy Class,
Data Class, and Speculative Generality.
• Comments(Excessive or Redundant) —Comments that describe what the code does rather than why,
repeating what clear naming already communicates. They become a maintenance burden.
Commentsare sometimes called the deodorant of code — masking a bad smell instead of removing
it. Treatment: Rename methods and variables to be self-documenting; remove redundant comments.
• Duplicate Code—Thesamelogicappears in twoormore places. Fixing a bug in one copybut not the
others introduces subtle inconsistencies. Treatment: Extract Method and call it from both sites.
• LazyClass —Aclassthat does almost nothing, typically just delegating to one other class. It adds
navigation overhead with no benefit. Treatment: Inline Class or Collapse Hierarchy.
• DataClass—Aclasswithonly fields, getters, and setters, no meaningful behaviour. Objects should
encapsulate both data and behaviour. Treatment: Move Method to bring behaviour to the data.
• Speculative Generality — Code written for hypothetical future requirements that do not exist yet
(gold plating). Unused abstract classes, hook methods, or parameters. Treatment: Collapse Hierarchy,
Inline Class, or remove unused code.

Category 5: Couplers [3 Marks]
Couplers represent excessive or inappropriate coupling between classes. The five coupler smells are:
Feature Envy, Inappropriate Intimacy, Message Chain, Middleman, and Incomplete Library Class.
• FeatureEnvy—Amethodthatspends moretimeusing data ormethodsofanother class than its
own. The intelligence is in the wrong place. Treatment: Move Method to the class whose data it
envies.
• Inappropriate Intimacy — Two classes that know too much about each other's private details, often
because a class was split incorrectly. Treatment: Move Method/Field, or Encapsulate Field to
establish proper boundaries.
• MessageChain—Aclientcalls object A to get B, then B toget C, then C to get D. The client is tightly
coupled to the entire chain. Treatment: Hide Delegate or extract the chain into a dedicated method.
• Middleman—Aclasswhoseonlypurpose isto delegate every call to another class. If more than half
of its methods simply forward to another object, it adds no value. Treatment: Remove Middleman
and call the delegated object directly.
• Incomplete Library Class — A library class that almost does what you need but is missing one or two
methods, forcing you to add logic in client code or subclass just to add the missing behaviour.
